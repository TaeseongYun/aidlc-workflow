# PM — Dependency-Aware Score Loop (의존성 인지 점수 루프)

> Product Requirements Document
> 생성: 2026-06-23 · 대상 프로젝트: `aidlc-workflow`
> 작성 방식: `/ouroboros:pm` Socratic interview (PM MCP 환경 이슈로 동등 산출물을 수기 작성)

---

## 1. 한 줄 요약 (Elevator Pitch)

"코드 구현 단 한 번의 요청"만으로, 의존성과 여러 검증 축을 **자동으로 반복 채점**하고, **100점 만점 중 85점을 넘겼을 때에만 온전히 종료**되는 루프 엔지니어링 기능을 aidlc-workflow에 추가한다. 사용자가 매번 "검증해줘"를 반복 요청할 필요가 없다.

---

## 2. 문제 정의 (Problem)

### 2-1. 현재 상태

aidlc-workflow는 **"무엇을 만들 것인가(What)"** 를 책임지는 워크플로우다. 다음 자산이 이미 존재한다.

- `core/readiness-score.md` — 구현 **이전**의 requirements 준비도를 100점 만점으로 채점 (80점=READY). **구현 후 검증용이 아니며, 사람이 GATE 통과 직전 1회 산출한다.**
- `templates/component-dependency.md` — 컴포넌트 간 의존성 매트릭스 템플릿 (순환 의존 점검 포함).
- `docs/stop-conditions.md` — GATE/점수 기반 STOP decision tree (모두 **구현 진입 전** 게이트).
- `docs/omc-ouroboros-integration.md` — **"어떻게 자동으로 굴릴 것인가(How)"** 는 OMC `ralph`/`autopilot` 또는 Ouroboros `evolve`로 **외부 위임**한다.

### 2-2. 갭 (Gap)

| 차원 | 현재 | 요구되는 것 |
|------|------|-------------|
| 점수 산출 시점 | 구현 **전** 1회 (readiness-score) | 구현 **후** 반복적·자동 |
| 점수 축 | 요구사항 준비도 6~8개 영역 | **의존성 해결 / 빌드 / 테스트 / AC 충족** 4축 |
| 의존성 | 설계 단계 매트릭스(1회 작성) | 각 피처/모듈에 **상주하는 검증용 md**, 해결 여부를 반복 채점 |
| 종료 판정 | 사람이 GATE 승인 | **85점 초과 시 자동 종료**, 점수 정체 시 자동 중단 |
| 트리거 | 매번 사람이 검증 요청 | **단 한 번의 구현 요청**으로 루프가 자율 반복 |

> 외부 도구(ralph/evolve)는 "UOW Verification 명령이 통과하는가"만 종료 조건으로 쓴다. **의존성 md를 기반으로 한 다축 점수 → 85점 임계 자동 완료**라는 개념은 워크플로우 어디에도 없다. 이 PRD가 그 공백을 채운다.

### 2-3. 사용자 고통 (User Pain)

- 구현 후 "의존성 다 됐어?", "검증 다시 해줘"를 매 라운드 수동으로 요청해야 한다 → **번거롭고 일관성이 깨진다.**
- "어디까지 됐는지"가 주관적이다 → **완료 여부를 객관적 수치로 말할 수 없다.**
- 의존성이 여러 종류(기능 선후관계 / 빌드·라이브러리 / 모듈 간)인데, 어디가 막혔는지 **한눈에 추적할 메커니즘이 없다.**

## 3. 목표 (Goals) & 비목표 (Non-Goals)

### 3-1. Goals

1. 의존성이 필요한 **각 피처/모듈 디렉토리 안**에 검증용 md 파일을 둔다 (분산 배치, 중앙 모음 아님).
2. 그 md를 기준으로 **4축 100점** 검증을 자동 채점한다.
3. **85점 초과** 시에만 "온전히 종료(완료)"로 판정하고 사용자에게 결과로 알린다.
4. 구현 직후부터 **반복적으로** 재검증하며, **이전 점수와 비교**한다.
5. **한 번의 구현 요청**으로 위 루프 전체가 자율 동작한다 (사용자 추가 개입 불필요).
6. 점수가 **정체(개선 없음)** 하면 자동 중단하고 사용자에게 보고한다.

### 3-2. Non-Goals (이번 범위 아님)

- **GATE(사람 승인) 자동 통과** — GATE-0~5는 여전히 사람만 통과시킨다. 이 루프는 GATE-3(구현 승인) **이후** 구간에서만 동작한다.
- 구현 **전** readiness-score(요구사항 준비도) 로직 변경 — 그대로 둔다. 이 루프는 **별도의 post-implementation 점수**다.
- 새로운 실행 엔진 작성 — 루프 자체는 기존 OMC `ralph`/Ouroboros `evolve` 또는 Claude Code 네이티브 루프에 **위임**하고, 이 기능은 그 위에 얹는 **"점수 루브릭 + md 규약 + 종료 판정"** 레이어다.
- 비즈니스 정책(환불/정산/권한 등) 자동 결정 — 정책 미확정은 여전히 STOP 조건이다.

## 4. 사용자 & 시나리오 (Personas & Scenarios)

### 4-1. 주요 사용자

- **단독 개발자** — S/M 사이즈 feature를 한 번 요청하고, 나머지는 루프가 알아서 85점까지 끌어올리길 원함.
- **팀 리드** — 여러 feature가 병렬 진행될 때 각 feature의 "현재 점수"로 완료 현황을 객관적으로 파악하길 원함.

### 4-2. 핵심 시나리오

**S1 — 한 번 요청, 자동 완료까지**
1. 사용자: "이 feature 구현해줘" (단 한 번).
2. 시스템: 구현 → 의존성 md 생성/갱신 → 4축 채점 → 점수 기록.
3. 85점 이하면: 부족한 축을 보완 구현 → 재채점 (반복).
4. 85점 초과: **"완료(85점 돌파)"** 를 결과로 사용자에게 보고하고 종료.

**S2 — 점수 정체로 중단**
1. 반복 채점 중 N회 연속 점수가 오르지 않음 (예: 의존성 한 축이 외부 요인으로 막힘).
2. 시스템: 루프 중단 → **"82점에서 정체. 막힌 축: 모듈 간 의존성(core-network API 미배포)"** 처럼 원인과 함께 사용자에게 보고.

**S3 — 다종 의존성 추적**
1. 한 feature가 (a) 선행 기능 완료, (b) `retrofit` 빌드 의존성, (c) `core-network` 모듈 API 세 가지에 의존.
2. 각 항목이 feature 디렉토리의 의존성 md에 체크 항목으로 존재.
3. 채점 시 어느 항목이 미해결인지 점수 분해로 드러남.

### 4-3. 엣지 케이스

- 의존성 항목이 **외부 요인(예: 아직 머지 안 된 다른 팀 PR)** 으로 영원히 해결 불가 → 정체 중단 + 명시적 사유 보고 (무한 루프 금지).
- 점수가 **하락**하는 경우 (회귀 발생) → 직전 라운드 대비 하락을 감지해 경고 (롤백 여부는 v1에서는 보고만, 자동 롤백은 향후 검토).
- 85점을 넘었으나 특정 **필수 축이 0점**인 경우 (예: 빌드는 0인데 다른 축 점수로 85 돌파) → **게이팅 규칙 필요** (아래 6-3 참조).

## 5. 핵심 개념 (Core Concepts)

### 5-1. 의존성 검증 md (Dependency Verification Doc)

- **위치**: 의존성이 필요한 **각 피처/모듈 디렉토리 안** (예: `aidlc-docs/features/<slug>/dependency-check.md` 또는 모듈 폴더 내).
- **내용**: 해당 위치가 의존하는 항목을 3종으로 분류해 체크리스트로 보유.
  - **기능 간 선후관계** — 예: "로그인 feature가 `completed` 상태여야 함".
  - **빌드/라이브러리 의존성** — 예: "`retrofit`, `hilt`, `compose-bom` 버전 설정 및 빌드 통과".
  - **모듈 간 의존성** — 예: "`feature-A`가 `core-network`의 API X를 올바르게 호출/구현".
- **상태**: 각 항목은 해결 여부(+근거)를 기록. 채점의 입력이 된다.

### 5-2. 4축 점수 루브릭 (Score Rubric, 100점)

| 축 | 의미 | 비고 |
|----|------|------|
| **의존성 해결 여부** | 위 md의 모든 의존성 항목이 충족되었는가 | 3종 의존성 합산 |
| **빌드/컴파일 통과** | 코드가 실제로 빌드되고 컴파일 에러가 없는가 | 객관적 검사 가능 |
| **테스트 통과 / 커버리지** | 관련 테스트가 통과하고 커버리지가 기준 이상인가 | 객관적 검사 가능 |
| **요구사항 / AC 충족도** | 원래 의도한 Acceptance Criteria가 충족되는가 | requirements/UOW 기준 |

- **각 축의 배점 비율: 25 / 25 / 25 / 25 (확정)**. 의존성 25 · 빌드 25 · 테스트 25 · AC 25.
- **채점 주체**: **LLM 자체 채점(자율)**. 단, 빌드·테스트 축은 LLM이 명령 실행 결과(객관 신호)를 근거로 채점하도록 강제한다 (환각 방지).

### 5-3. 점수 기반 종료 (Score-Gated Termination)

- **완료 조건**: `score > 85` → "온전히 종료(완료)"로 판정, 결과 보고 후 루프 종료.
- **중단 조건**: 점수가 **2회 연속(확정)** 개선되지 않으면 → 정체로 판단, 중단 + 원인 보고.
- **안전장치 (확정)**: 최대 **10회 반복 또는 30분** 중 먼저 도달 시 중단 (무한 루프 방지).

## 6. 요구사항 (Requirements)

### 6-1. 기능 요구사항 (Functional)

- **FR-1** 시스템은 의존성이 필요한 각 피처/모듈 디렉토리에 의존성 검증 md를 생성·갱신한다.
- **FR-2** 의존성 md는 3종 의존성(기능 선후 / 빌드·라이브러리 / 모듈 간)을 체크 항목으로 분류해 보유한다.
- **FR-3** 시스템은 4축(의존성/빌드/테스트/AC)으로 100점 만점 점수를 산출한다.
- **FR-4** 빌드·테스트 축은 실제 명령 실행 결과를 근거로 채점한다 (LLM 단독 추정 금지).
- **FR-5** 시스템은 매 라운드 점수를 기록하고 직전 라운드와 비교한다.
- **FR-6** `score > 85`이면 "완료"로 판정하고 결과를 사용자에게 보고한 뒤 종료한다.
- **FR-7** 점수가 정체(N회 연속 미개선)하면 루프를 중단하고 막힌 축/항목을 사용자에게 보고한다.
- **FR-8** 단 한 번의 구현 요청으로 위 루프 전체가 추가 사용자 개입 없이 자율 반복된다.
- **FR-9** 루프는 GATE-3(구현 승인) 통과 이후 구간에서만 동작하며, GATE를 자동 통과시키지 않는다.

### 6-2. 사람이 확정해야 할 결정 (Open Items / Decide-Later)

> 모든 Open Item이 기본값으로 확정되었다 (사용자 승인 완료). 요구사항 단계에서 그대로 사용한다.

- ~~**OI-1 (배점 비율)**~~ → **확정: 25/25/25/25** (의존성/빌드/테스트/AC 균등).
- ~~**OI-2 (정체 판정 N)**~~ → **확정: 2회 연속 미개선 시 정체**.
- ~~**OI-3 (상한)**~~ → **확정: 최대 10회 반복 또는 30분** (둘 중 먼저 도달 시 중단, 무한 루프 방지).
- ~~**OI-4 (회귀 시 동작)**~~ → **확정: v1은 보고만** (점수 하락 감지 시 경고만, 자동 롤백 없음).
- ~~**(실행 엔진)**~~ → **확정: OMC `ralph` 기본** (S/M feature에 적합, `omc-ouroboros-integration.md` §2-2 핸드오프 경로 재사용). 측정 가능 목표가 강한 경우 Ouroboros `evolve`로 대체 가능.

### 6-3. 게이팅 규칙 (필수)

- **GR-1** 85점 돌파라도 **빌드 축이 0점이면 완료로 판정하지 않는다** (빌드 실패 코드를 "완료"로 부르지 않음 — 프로젝트 룰 "빌드 깨진 채 커밋 금지"와 일치).
- **GR-2** BLOCK으로 분류된 미해결 의존성이 1건이라도 있으면 의존성 축은 상한 점수로 제한 (readiness-score의 BLOCK 규칙과 동일한 철학).

### 6-4. 비기능 요구사항 (NFR)

- **NFR-1 (충돌 방지)** 상태 디렉토리 분리 규칙 준수 — `aidlc-docs/`(SoT), `.omc/state/`, `.ouroboros/`를 덮어쓰지 않는다 (`omc-ouroboros-integration.md` §4-1).
- **NFR-2 (audit append-only)** 루프 이벤트를 audit에 기록할 때 `[LOOP]` 등 프리픽스로 구분, 기존 엔트리 미수정.
- **NFR-3 (투명성)** 점수는 숫자만 쓰지 않고 **근거와 함께** 기록한다 (readiness-score 규칙 계승).
- **NFR-4 (정직한 종료)** "완료"는 오직 검증 통과 시에만 보고한다. 테스트 실패/스킵 시 그 사실을 명시한다.

## 7. 구현 형태 (Delivery Form)

> 인터뷰 결정: **기존 루프 확장 + 독립 슬래시 명령 혼합 (1+2)**.

- **A. 기존 루프 확장** — `omc-ouroboros-integration.md`의 ralph/autopilot/evolve 핸드오프 프롬프트에 "의존성 md + 4축 점수 + 85 임계 종료" 규약을 주입한다. 즉 ralph의 종료 조건을 "UOW Verification 통과"에서 "**의존성-인지 4축 점수 > 85**"로 확장 가능하게 한다.
- **B. 독립 슬래시 명령** — 어느 프로젝트에서든 호출 가능한 재사용 명령 (예: 가칭 `/ctx-score-loop` 또는 `/score-loop`). 단독으로 의존성 md 생성 → 채점 → 루프 → 보고를 수행.

### 7-1. 신규/변경 산출물 (예상)

- `core/dependency-score.md` (신규) — 4축 100점 루브릭 + 게이팅 규칙 정의 (readiness-score와 형제 문서).
- `core/dependency-score.schema.yaml` (신규) — 프로그래매틱 채점용 스키마.
- `templates/dependency-check.md` (신규) — 피처/모듈에 상주하는 의존성 검증 md 템플릿 (3종 의존성 체크리스트 + 점수 테이블).
- `docs/score-loop-guide.md` (신규) — 루프 동작/종료 조건/핸드오프 가이드.
- `docs/omc-ouroboros-integration.md` (변경) — ralph/evolve 핸드오프에 점수 루프 종료 조건 추가.
- `templates/feature-status.md` (변경) — post-implementation score 추적 행 추가.
- 슬래시 명령 스킬 (신규) — `skills/<name>/` + `scripts/install-skills.sh` 등록.

## 8. 성공 지표 (Success Metrics)

- 사용자가 구현 요청 **1회** 후 추가 "검증해줘" 요청 **0회**로 완료에 도달하는 비율.
