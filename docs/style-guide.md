# Style Guide

산출물 작성 시 한글/영어 혼용 기준이다.

## 원칙
- 산출물은 팀원이 빠르게 읽을 수 있어야 한다.
- 고유 명사와 기술 용어는 영어를 유지한다.
- 설명과 판단은 한글로 작성한다.

## 섹션 헤딩

| 위치 | 언어 | 예시 |
|------|------|------|
| 템플릿 섹션 제목 | 영어 | `## Goal`, `## In-Scope`, `## Summary` |
| 워크플로우 규칙 문서 섹션 | 한글 | `## 원칙`, `## 게이트 목록`, `## 승인 필요 항목` |

이유: 템플릿 헤딩은 프로젝트 간 일관성을 위해 영어로 고정한다. 규칙 문서는 읽는 대상이 팀 내부이므로 한글이 자연스럽다.

## 필드 라벨

| 위치 | 언어 | 예시 |
|------|------|------|
| 질문 문서 라벨 | 한글 (필수) | 분류, 영향도, 이유, 선택지, 미응답 시, [답변] |
| UOW 필드 라벨 | 한글 | 책임, 예상 위치, 의존성, 규모, 수용 기준, 검증 방법 |
| 상태 값 | 영어 | `OPEN`, `ANSWERED`, `BLOCK`, `ASSUME-A`, `TODO` |

이유: `common/question-rules.md`의 "라벨은 반드시 한글" 규칙과 일치시킨다. 상태 값은 코드/자동화에서 파싱 가능하도록 영어를 유지한다.

## 본문

| 항목 | 언어 | 예시 |
|------|------|------|
| 요구사항 설명 | 한글 | "재구매 고객에게 자동으로 할인 쿠폰을 발급한다" |
| 기술 용어 | 영어 유지 | API, DB, Entity, Repository, JPA, batch |
| 파일 경로 | 영어 | `ctx/project-profile.ctx.md` |
| 커밋 메시지 | 한글 (type/scope 제외) | `feat: (coupon) 쿠폰 도메인 기본 구조 추가` |

## 혼용 금지 사항
- 하나의 문장 내에서 한글과 영어를 불필요하게 섞지 않는다.
  - Bad: "유저의 order를 cancel하는 로직"
  - Good: "사용자의 주문을 취소하는 로직" 또는 기술 문맥에서 "Order 취소 로직"
- 같은 문서 내에서 동일 개념을 한글/영어로 번갈아 쓰지 않는다.
  - Bad: "쿠폰"과 "coupon"을 같은 문서에서 혼용
  - Good: 하나로 통일 (본문에서는 "쿠폰", 코드/경로에서는 "coupon")
